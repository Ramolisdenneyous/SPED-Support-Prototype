# SPED Support Swarm — Pilot / Demonstration Prototype Build Brief


## Purpose


Build a working pilot demonstration prototype for an AI-assisted SPED classroom support system.


This prototype is intended for presentation to school administrators, district IT, SPED leadership, and potentially a school board. It should demonstrate the core product concept without using real student data.


The system should show how a teacher-facing dashboard could monitor student engagement, assignment progress, and support needs across a classroom using simulated student data.


The prototype must be designed so it can later integrate with Google Classroom, Google Workspace Admin APIs, Chrome Management APIs, and Gemini/Gemini Gems, but the first version should run entirely from artificial demo data.


---


## Core Concept


The app has two student-side systems that both report to the teacher:


### 1. Monitoring Minder System (prototype goal)


This system passively observes simulated student status data.


It does not tutor students.


Its job is to:
- monitor task progress
- identify confusion
- detect off-task behavior
- detect possible disengagement
- flag students who may need help
- send compact status updates to the teacher dashboard


### 2. Tutor / Accommodation Agent System (MK2 Goal)


This system represents a future student-facing AI tutor.


In the prototype, this may be simulated rather than fully implemented.


Its job is to eventually:
- provide assignment help
- chunk tasks
- support reading accommodations
- provide executive-function support
- avoid answer dumping
- follow student-specific accommodation rules


For the first demonstration prototype, tutoring should be represented as a separate subsystem from monitoring.


The monitoring system watches behavior.
The tutor system interacts with students.
Both send useful information to the teacher.


Do not merge these systems. (although for the prototype we will attempt to offload some of these tasks to the Teacher side AI tool)

Frontend Requirements
Build a teacher-facing dashboard optimized for tablet display in Chrome.
The dashboard should show a classroom of 10 simulated students.
Preferred visual metaphor:
* isometric classroom or top-down classroom
* each student represented by a small “minder” avatar
* students themselves do not need to be visually displayed
* minder avatar state communicates student status at a glance
Each minder should have a visible state such as:
* Working
* Confused
* Idle
* Researching
* Off Task
* Using Accommodation
* Needs Help
* Escalation Recommended
The teacher should be able to click or tap a student/minder and open a detail panel showing:
* current status
* recent activity
* assignment progress (with a full list of current assignments) 
* confusion level
* engagement level
* Approved accommodation (clicking these will give implementation options) 
* minder summary
* tutor summary placeholder
* recent simulated events
The teacher dashboard should update live without requiring page refresh.
Use WebSockets or server-sent events if appropriate. Polling is acceptable for the first prototype if simpler.
________________


Backend Requirements
The FastAPI backend should act as the source of truth for:
* students
* simulated assignments
* minder states
* event logs
* dashboard summaries
* alert state
* teacher-facing status payloads
The backend should expose REST endpoints for:
GET /health
GET /students
GET /students/{student_id}
GET /students/{student_id}/events
GET /dashboard/state
POST /simulator/tick
POST /simulator/reset




Optional real-time endpoint:
WS /dashboard/live
The backend should store state in PostgreSQL.
Use SQLAlchemy or SQLModel if helpful.


We must try to keep active states in the backend as much as possible, Ideally the fron’t end will be stateless, and only report the current states read from the back end.


Make sure to build logs for everything that could break, as we will be primarily using logs to troubleshoot issues as we build.

Simulator Service
Create a separate simulator service responsible for generating artificial student behavior.
The simulator should create 10 fake students.
Each student should have a unique behavior profile with different distributions.
Use 3d6-style rolls to create roughly normal behavior distributions.
Example student profiles:
{
  "student_id": "demo_01",
  "display_name": "Student 01",
  "profile_type": "steady_worker",
  "focus_baseline": 13,
  "confusion_baseline": 8,
  "help_seeking": 10,
  "off_task_tendency": 6,
  "reading_support_needed": false,
  "executive_function_support": true
}


Create at least 10 profile archetypes:
1. Steady worker
2. High achiever but low help-seeking
3. Easily confused but persistent
4. Frequent task-switcher
5. Strong reader, poor task completion
6. Dyslexia support profile
7. Executive-function support profile
8. Anxiety/frustration spike profile
9. Off-task / gaming tendency
10. Quietly stalled student
Each simulator heartbeat should:
* roll 3d6 per student (this gives us a normal distribution
* Most likely actions clustered in the center (for example, working on a roll of 10 or 11)
* Escalation Recommended should be rare, place it on a roll of 18 only.
* apply student-specific modifiers
* update status
* generate event logs
* produce a minder summary
* create alert flags when thresholds are crossed
Heartbeat interval: (students are staggered, and rotate alphabetically, with 10 students at 50sec we have one student check in every 5sec).
* configurable by environment variable
* default: 50 sec seconds for demo mode
Do not call an AI model once per student per minute in the first version. In fact we may not have the minder agents sending active updates, just updating the local status when events occur so most updates to the teacher are just backend apps communicating with each other.
Use deterministic/rule-based simulated minder logic first.

UX / Presentation Goal
The presentation should make the value obvious in under 60 seconds.
The administrator should immediately understand:
“This helps a SPED teacher see which students are working, which are stuck, which are off task, and which need intervention.”
The dashboard should avoid technical clutter.
Prioritize:
* visual clarity
* fast status recognition
* simple teacher actions
* obvious student alert states
* smooth demo flow
Suggested presentation flow:
1. Teacher opens dashboard.
2. Ten minder avatars are visible.
3. Most are working.
4. One student becomes confused.
5. One student drifts off-task.
6. One student uses an accommodation successfully.
7. One student escalates to “Needs Help.”
8. Teacher taps the student.
9. Detail panel shows recent events and recommendation.
10. Teacher sees how this scales classroom awareness.
________________


Important Safety / Messaging Requirements
The prototype must not imply:
* autonomous grading
* disciplinary surveillance
* replacement of teachers
* automated IEP decision-making
* use of real student data
* uncontrolled AI access for minors
The prototype should imply:
* teacher amplification
* classroom awareness
* SPED support
* executive-function support
* accommodation visibility
* early intervention
* human-in-the-loop design
Use language like:
teacher-facing support
guided intervention
student progress awareness
accommodation support visibility
early warning indicators
teacher-supervised AI assistance
Avoid language like:
surveillance
behavior scoring
automated discipline
AI replaces aides
AI decides accommodations

Success Criteria
The prototype is successful when:
* the app runs locally with Docker Compose
* the app can deploy to Railway
* the dashboard shows 10 simulated students
* student statuses update automatically
* minder events are generated and stored
* teacher can inspect individual student details
* alert states are visually obvious
* no real student data is required
* Google/Gemini integration points are documented but not required
* the demo clearly communicates the SPED teacher support concept
________________


Non-Goals For First Prototype
Do not build these yet:
* real Google Classroom OAuth integration
* real student data ingestion
* real Gemini tutor conversations
* real IEP storage
* real Chromebook telemetry
* grading systems
* parent communication systems
* administrative discipline features
These are future phases after the presentation prototype is stable.
________________


Final Instruction
Build this as a clean, modular prototype.
Prioritize demo clarity over architectural perfection.
The goal is to create a working proof-of-concept that helps administrators understand the vision:
A teacher-supervised AI support layer that helps SPED teachers monitor classroom progress, identify student needs earlier, and eventually provide safe accommodation-aware tutoring support.
